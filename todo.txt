AUTHENTICATION & CLOUD — current architecture (as of this session)
──────────────────────────────────────────────────────────────
One account, mandatory, database-primary. No multi-account switcher, no guest
mode, no push/pull sync engine — all of that was built and then deliberately
torn back out this session in favor of something much simpler:

- src/services/storage/localEnvelope.ts — the single primitive every domain
  file in services/storage/local/ goes through. Reads/writes go straight to
  Supabase's `user_data` table (RLS-scoped to auth.uid()) for whoever's signed
  in; AsyncStorage is purely a write-through cache (instant app open, tolerates
  brief offline use), never the source of truth, never needs conflict
  resolution. Covers every domain named in the app's own words: Routine, Diet,
  Bloodwork, Cal Burn, Photos (metadata only), Progress, Account/profile,
  Targets. Not premium-gated — every signed-in user gets this, any tier.
- src/services/storage/cloud/photoCloud.ts — the one exception: actual photo
  *bytes* upload only for premium/ultra (see PhotosScreen.tsx), automatic
  based on tier, no manual toggle. Free tier images stay device-only.
- src/services/auth/authService.ts — Supabase email/Google/phone sign-in.
  Google needs an OAuth Client ID registered in Supabase Dashboard →
  Authentication → Providers → Google. Phone needs an SMS provider (Twilio /
  MessageBird / Vonage / TextLocal, your choice) under Authentication →
  Providers → Phone (paid, per-message). Neither tested against a live
  provider yet — code-complete, untested until that dashboard setup is done.
- src/app/providers/AuthProvider.tsx — session context; gates rendering until
  the initial session check resolves; forces a clean remount on sign-out (not
  on the very first sign-up, so mid-onboarding progress survives).
- src/app/navigation/RootNavigator.tsx — isAuthenticated is the gate for
  showing Main at all. src/app/navigation/OnboardingNavigator.tsx starts at
  Login directly (skipping Language/Welcome) if this device has a cached
  profile from before, i.e. "kick out to the login screen" after signOut().
- src/modules/onboarding/screens/LoginScreen.tsx — on success, checks
  loadUserProfile() (reads straight from the DB) and skips straight to Main
  for an account that already has one, instead of re-running Welcome →
  GoalDescription → OnboardingChat → ... again.
- src/app/providers/SubscriptionProvider.tsx — reads tier from `entitlements`
  when signed in, falls back to local AsyncStorage tier otherwise (dev
  override). Unrelated to the storage model above — entitlements is about
  subscription tier, not data persistence.
- supabase/schema.sql — entitlements + user_data tables, RLS, storage bucket.
  Already fits the current model as-is, no schema changes needed. Run
  manually once in the Supabase SQL editor (not executed automatically).
  Connectivity-verified against a real project earlier this session.
- shared/components/Input.tsx (LabeledInput) — implemented, moved out of
  LoginScreen.tsx.

Deliberately removed this session (built, then reverted as overcomplicated):
multi-account switcher + SecureStore token cache (knownAccountsStorage.ts,
accountSessionStore.ts), the push/pull/conflict-resolution sync engine
(syncRegistry.ts, syncService.ts), the dataSync/photosSync toggle preferences
(syncPreferencesStorage.ts), and a brief "Guest is a usable mode" experiment.

Still needed to make this live: create the actual Supabase project (Mumbai
region) if not already done, run supabase/schema.sql, set
EXPO_PUBLIC_SUPABASE_URL / EXPO_PUBLIC_SUPABASE_ANON_KEY in .env. Not yet
tested end-to-end against a real signed-in session doing real reads/writes —
needs the app actually running against a live project.

Still pending, not started:
- RevenueCat / Play Store / App Store IAP → write to `entitlements`
- Razorpay/UPI payment flow → write to `entitlements` (same table, so either
  path works without touching SubscriptionProvider again)
- Calory burn's per-day keys (e.g. 'calburnActs_2026-07-01') work fine under
  the new direct read/write model (each day is just its own `domain` string)
  — no special-casing needed, unlike the old sync-registry design.
- Restore-on-new-device for photos: photoCloud.ts can list what's in the cloud
  but doesn't re-download missing photos into local files yet.
- Offline write queue — DONE. A failed cloud write (offline) still lands in the
  local cache immediately (UI never sees a failure) and gets queued in
  localEnvelope.ts's pendingWrites. Drains automatically on the next successful
  cloud read/write or when the app returns to the foreground (AppState). Queue
  entries are tagged per-userId so a pending write from account A never gets
  pushed under account B's session. No retry backoff/limit — simple by design.
